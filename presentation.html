01 · Problem Statement
Workforce insights are too slow, too manual, too siloed
Every data question becomes a ticket. Every chart takes days. Every manager is blocked.
📋 Reporting Bottleneck
  • Ad-hoc spend questions require SQL engineers — 2–5 day turnaround
  • Monthly workforce reports are manually assembled from multiple BigQuery exports
  • Business leaders make decisions on stale snapshots, not live data
📦 Data Siloed & Opaque
  • Workforce, vendor, spend and FTE data lives across 3 environments (DEV / UAT / PRD)
  • No single view — managers switch between BigQuery, Excel and email threads
  • Schema knowledge locked inside a few engineers' heads
🔐 Access & Governance Friction
  • Tableau / PowerBI licenses are expensive and require analyst training
  • Granting direct BigQuery access to non-technical users is a security risk
  • No per-user audit trail for data access
🔄 No Self-Service Analytics
  • "Top 10 vendors by offshore spend this quarter" — requires a Jira ticket
  • Dashboard requests pile up in the BI team's backlog
  • Non-technical PMs can't explore data independently
⚠️
Business Impact
Delayed workforce decisions cost time and budget. When capital vs. expense classifications are wrong by even one month, it affects financial reporting across the entire portfolio.

04 · Agent Interaction
How the AI agent answers a question end-to-end
Two paths: a fast conversational path (~1s) and an agent loop with live BigQuery tool calls (~3–6s).
⚡ Fast Path ~1s · No BigQuery call · Conversational & definitional questions
1
User message arrivese.g. "What does YTD mean?" / "Explain capital vs expense"
2
Intent Classifier (0ms)Regex check — no analytical keywords → route to fast path
3
Single Gemini callchat_turn() — no tools attached, schema context injected, glossary applied
4
JSON response → SSE{ text, intent:"explain", suggested_questions } streamed immediately
🤖 Agent Path 3–6s · Live BQ queries · Data, chart & analysis questions
1
Intent Classifier → Agente.g. "Show me top 10 vendors by YTD spend" → analytical keywords detected
2
Contextual Hints (parallel)Matching keyword columns fetched concurrently via ThreadPoolExecutor — cached 10 min
3
Agent Loop (max 4 rounds)Gemini reasons → calls run_bigquery_query tool → BQ executes → result back to Gemini
4
Status events stream to UI"Analyzing…" → "Running 2 queries in parallel (step 1)" → "Formatting…"
5
Final JSON + widget dataRoute serves { text, intent, widget{sql,chart_type,data}, suggested_questions } — BQ result reused from cache
Agent Tool: run_bigquery_query
The only tool the agent has is run_bigquery_query(sql). Gemini can call it multiple times per turn — first to discover filter values (SELECT DISTINCT), then to fetch the actual data. Results are returned to Gemini as structured JSON. All tool calls within a round run in parallel via asyncio.gather.
Tool calls parallelized 60s result cache Max 4 BQ round-trips

08 · Security & Authorization
Defense-in-depth: user-scoped credentials at every layer
No user ever gets broader BigQuery access than their Google identity already has.
Authentication Flow
1
Browser sends Bearer tokenGoogle OAuth2 access token in Authorization header on every API request
2
Backend validates tokenCalls Google's /userinfo endpoint — extracts user ID + email. Falls back to anonymous if token absent (dev mode).
3
Token forwarded to BigQuerySame Bearer token wraps into google.oauth2.Credentials — BQ runs queries under the user's own GCP IAM identity
4
User-scoped data isolationGlossary terms and favorites in SQLite are filtered by user ID — no cross-user data leakage
Security Controls
🛡️
CORS Whitelist
Allowed origins explicitly listed: localhost:5173, localhost:3000, and CLOUD_RUN_URL env var. No wildcard (*) in production.
🔒
No Direct SQL from Users
All SQL is AI-generated from natural language. Only admins can use the /api/query/sql endpoint. Users cannot construct arbitrary queries.
📝
BigQuery Job Audit Trail
Every BQ query runs as a job attributed to the user's GCP identity — full audit log in Cloud Logging automatically.
🔑
Credential Hierarchy
Request token → env GOOGLE_OAUTH_TOKEN → Service Account JSON → Application Default Credentials (ADC). Vertex AI uses separate SA fallback chain.
Compliance Notes
  • No PII stored in application database — only user IDs and term definitions
  • BigQuery row-level security can be applied at the GCP layer independently
  • All secrets (API keys, tokens) via .env / environment variables — never hardcoded
